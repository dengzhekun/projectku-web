## 商品数据从哪里来
当前项目的真实商品基础数据来自后端数据库和商品接口，不应该只依赖知识库。

初始化 SQL 中商品覆盖这些类目：
- 手机类，商品 ID 1-20。
- 电脑和外设类，商品 ID 21-40。
- 家电类，商品 ID 41-60。
- 数码配件类，商品 ID 61-80。
- 家居收纳类，商品 ID 81-100。
- 美妆个护类，商品 ID 101-120。
- 食品生鲜类，商品 ID 121-140。
- 运动户外类，商品 ID 141-160。
- 补充商品，商品 ID 161-170。

客服遇到商品价格、库存、SKU、有没有货、有哪些产品这类问题，应优先调用后端商品实时接口。只有商品接口查不到时，才回退到知识库。

## SKU 规格数据
当前初始化 SQL 中，iPhone 15 Pro 128G 有两个 SKU：
- 黑色 + 128G，价格 7999.00，库存 50。
- 白色 + 128G，价格 7999.00，库存 50。

如果用户只问“苹果15 Pro 多少钱”，但没有说颜色或容量，客服可以先给出查到的相近商品，并追问具体规格。

如果用户问“苹果15”而不是“苹果15 Pro”，商品工具应优先匹配 iPhone 15 128G，不应误识别成 iPhone 15 Pro。

## 库存什么时候扣减
当前项目不是加入购物车时扣库存，而是提交订单时扣库存。

下单时，后端会对购物车中勾选商品执行库存扣减：
- 如果有 SKU，先扣 SKU 库存。
- 再扣商品基础库存。
- 扣减语句带有 `stock >= quantity` 条件。
- 如果库存不足，提交订单失败并返回库存不足。

这属于数据库层面的条件更新，可以避免两个用户同时购买最后一件时都成功扣成负库存。

待支付订单取消时，系统会恢复订单商品库存。

支付成功时，系统不会再次扣库存，而是增加商品销量。

## 加入购物车和库存
加入购物车不等于锁定库存。购物车里的商品在结算时仍会按最新商品价格和库存校验。

如果用户问“我加入购物车后库存会减少吗”，回答：不会。当前系统是在提交订单时扣库存；如果订单取消且仍是待支付状态，会恢复库存。

## 商品评价数据
当前项目有评价表和评价接口：
- 发布评价：`POST /v1/reviews`
- 查询商品评价：`GET /v1/reviews?productId=...`
- 查询我的评价：`GET /v1/reviews`

评价发布校验：
- 必须有订单 ID。
- 必须有商品 ID。
- 评分必须在 1 到 5 之间。
- 订单必须属于当前登录用户。

当前 `back/sql/seed_reviews.sql` 中为商品 ID 1-170 准备了评价种子数据，多数商品每个 2 条评价，评价内容包含质量、包装、物流、发货速度、实物描述一致性等反馈。

客服回答商品口碑时可以引用“当前商品评价列表”或“商品详情页评价”，不要凭空编造某个商品评价数量或好评率。需要精确评价内容时，应查询商品评价接口。

## 商品问答边界
价格、库存、SKU、商品清单问题：
- 优先查后端商品接口。
- 查到商品后基于真实商品数据回答。
- 回答末尾加“实际价格和库存以下单页为准”。

商品接口查不到时：
- 再走知识库检索。
- 如果知识库也没有明确资料，应说明暂未查询到该商品。

不要把“苹果”单独当成手机；但“苹果15”“苹果15 Pro”“iPhone 15”可以确定为手机商品查询。
