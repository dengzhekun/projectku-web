# RAG / 知识图谱方案评估与项目建议

日期：2026-04-26

## 1. 结论

结合当前项目现状，推荐顺序如下：

1. 优先采用 `商品实时查询工具 + 现有 RAG / 可升级为 LightRAG + 歧义追问 + 可选联网工具` 的混合架构。
2. 在开源知识库检索方案中，优先考虑 **LightRAG** 作为下一步升级方向。
3. **GraphRAG** 适合作为二期增强方案，不建议当前阶段直接重构为主链路。
4. **llm-wiki** 更适合内部知识整理和研究沉淀，不适合作为线上 AI 客服主检索引擎。

一句话判断：

- 如果目标是尽快提升线上 AI 客服效果，优先选 **LightRAG**。
- 如果目标是做重型、多跳、全局知识发现，后续再评估 **GraphRAG**。
- **llm-wiki** 不建议直接作为客服检索核心。

## 2. 当前项目现状与核心问题

当前项目已经具备：

- Java / Spring Boot 后端
- Python AI service
- 商品实时接口
- 知识库 / 向量检索基础能力
- 管理后台

当前真正需要优化的核心问题不是“是否换一个最先进的名词”，而是：

1. 商品别名识别不稳定，例如“苹果15”“iPhone15”“苹果15手机”。
2. 型号 / 容量 / 规格不完整时，客服不会稳定追问。
3. 商品实时价格、库存、SKU 与知识库回答需要共存。
4. 站内知识、商品实时数据、外部实时信息三类问题需要不同处理链路。
5. 目前 agent 的路由、置信度和回退策略还不够结构化。

## 3. 三个方案的适配判断

## 3.1 LightRAG

推荐级别：高

适合原因：

- 更偏工程落地，接入成本相对低。
- Python 生态友好，适合接到现有 AI service。
- 同时利用图结构与向量检索，适合电商知识问答。
- 支持 API / Server / WebUI，更适合服务化接入。
- 支持增量更新，方便后续持续补文档、商品知识、售后规则。

适合本项目的场景：

- 售后政策问答
- 商品参数知识问答
- 品牌 / 系列 / SKU 关系问答
- 多跳关联问题，例如“这款手机支持哪些配件”“哪类商品享受哪些政策”

技术栈需求：

- Python
- 推荐使用 `uv`
- 可通过 Docker / Docker Compose 部署
- 可搭配 PostgreSQL、Neo4j、OpenSearch 等存储
- embedding 可用 `BAAI/bge-m3`
- reranker 可考虑 `BAAI/bge-reranker-v2-m3`

## 3.2 GraphRAG

推荐级别：中

适合原因：

- 更适合复杂多跳推理、全局主题总结、知识社区摘要。
- 对大型文档集、复杂关系抽取有优势。
- 研究和分析能力更强。

不建议当前直接主用的原因：

- 体系更重，接入和维护复杂度更高。
- 索引过程更依赖 LLM，成本更高。
- 更适合复杂知识发现，不适合当前优先解决线上客服稳定性问题。

技术栈需求：

- Python 3.10-3.12
- `pip install graphrag` 或 `uv`
- 依赖 LLM 做实体、关系、社区摘要抽取
- 索引输出依赖 Parquet 和配置的向量存储
- 标准索引 / 快速索引两套模式

更适合的使用阶段：

- 站内知识量明显扩大后
- 需要复杂知识探索后
- 需要做更强的全局总结、专题知识图谱后

## 3.3 llm-wiki

推荐级别：低

适合原因：

- 更像“知识编译”和“内部 wiki 沉淀”方案。
- 适合把零散网页、说明文档、规则、研究资料整理成长期知识资产。

不适合当前主链路的原因：

- 不是典型的线上客服问答主检索引擎。
- 更适合作为内部研究 / 资料沉淀系统，而不是面向终端用户的实时问答核心。

适合的定位：

- 内部知识整理
- 团队文档沉淀
- 研究资料归档

## 4. 知识图谱能解决什么，不能解决什么

知识图谱 / graph-based RAG 可以明显帮助解决：

1. 商品别名归一化  
   例如：“苹果15”“iPhone 15”“iPhone15”“苹果15手机”归一到同一实体。

2. 商品层级和变体关系  
   例如：`iPhone 15 -> Pro / Plus / 128G / 256G`。

3. 品牌、型号、配件、政策之间的关系组织  
   例如：某型号支持哪些配件、适用哪些售后政策。

4. 多跳问题  
   例如：从品牌 -> 系列 -> 商品 -> SKU -> 售后策略的组合问答。

但是它不能单独解决以下问题：

1. 意图路由  
   需要区分“商品实时查询”“站内知识问答”“外部实时信息”。

2. 歧义追问  
   例如用户只问“苹果15pro多少钱”，没说容量，系统不能直接猜。

3. 置信度决策  
   不能只做 0 / 1 判断，更合理的是 high / medium / low 分级。

4. 实时数据优先级  
   商品价格、库存、SKU 必须优先查后端实时接口，不能让知识库瞎答。

5. 联网搜索治理  
   是否联网、何时联网、联网结果如何融合，需要单独的工具路由层。

结论：

知识图谱能提升知识组织和召回质量，但不能代替 agent 编排。

## 5. 对当前项目的建议架构

推荐采用四段式路由：

### 5.1 第一层：意图分类

把问题分为：

- 商品查询类
- 售后 / 规则 / 站内知识类
- 一般说明类
- 外部实时信息类

### 5.2 第二层：商品查询优先走实时工具

对商品查询类问题：

1. 先做商品名 / 型号 / 别名归一化
2. 调用后端商品实时接口
3. 若命中多个 SKU 或多个容量，主动追问
4. 命中唯一商品后，用真实价格 / 库存 / SKU 回答
5. 回答末尾加“实际价格和库存以下单页为准”

### 5.3 第三层：站内知识走 RAG / LightRAG

如果不是实时商品数据，而是：

- 售后政策
- 配送规则
- 商品知识说明
- 配件兼容
- 常见问题

则走知识库检索，可逐步从当前 RAG 升级到 LightRAG。

### 5.4 第四层：外部实时信息走联网工具

如果问题明显属于：

- 新闻
- 外部平台价格
- 第三方政策变化
- 实时行业信息

则再调用联网搜索工具，而不是要求知识库承载这类问题。

## 6. 为什么不是“只靠 prompt”

目前问题不能只靠一段大 prompt 解决，原因如下：

1. prompt 只能约束模型行为，不能替代真实工具调用。
2. prompt 不能替代后端商品 API。
3. prompt 不能稳定处理 SKU 歧义。
4. prompt 不能替代置信度评分和路由策略。
5. prompt 不能代替图谱、向量检索和联网搜索的职责分层。

正确方向应该是：

`规则路由 + 工具调用 + 检索 + 歧义追问 + 最终回答生成`

## 7. “模型能联网”与“agent 有联网工具”不是一回事

需要区分三件事：

1. 某个聊天产品界面支持联网
2. 某个 API 模型本身具备联网模式
3. 你的项目里的 agent 是否显式接入了联网工具

对项目工程来说，真正可靠的是第 3 种：

- 商品数据：走站内商品接口
- 站内知识：走知识库 / LightRAG
- 外部实时信息：走联网工具

只有这样，路由、审计、优先级和回退策略才可控。

## 8. 最终推荐

当前项目的最优落地顺序建议如下：

1. 保持“商品实时查询工具优先”的主路径
2. 保留原有知识库作为回退
3. 把知识库层升级目标定为 **LightRAG**
4. 给 agent 增加：
   - 歧义追问
   - 置信度分级
   - 外部联网搜索工具
5. GraphRAG 放在二期评估
6. llm-wiki 用于内部知识整理，而不是线上客服主问答链路

## 9. 一句话总结

这次优化方向不应该是“把所有问题都交给知识图谱”，而应该是：

**商品实时工具 + LightRAG + 歧义追问 + 联网工具 + 结构化路由**

这是比单纯加 prompt、更稳定也更符合当前项目形态的方案。
